付费收入波动的系统化根因定位方法。核心思路：先看整体趋势，再层层拆解，定位到具体维度和用户行为，最后排除外部因素得出可行动结论。

核心公式

收入 = DAU × 付费率 × ARPPU 付费率 = 付费用户数 / 活跃用户数 ARPPU = 付费金额 / 付费用户数 ARPU = 付费金额 / 活跃用户数 = 付费率 × ARPPU

任何收入变化，最终都能归因到 DAU、付费率、ARPPU 三个乘数之一（或组合）。先判断是哪个乘数在变，再往下拆。

6 步根因定位法

1整体趋势判断

先看收入大盘的时间趋势，确认是真实波动还是正常周期。对比环比/同比、节假日、版本节点。明确"变了多少、从什么时候开始变"。

2多维拆解贡献度

把收入变化拆到 DAU / 付费率 / ARPPU 三因子，算出各因子对总变化的贡献度，锁定主因子。

3交叉分析

沿主因子继续按维度下钻（国家、渠道、版本、新老用户、付费档位…），找出贡献最大的细分人群/场景。警惕辛普森悖论。

4行为归因

进入用户行为层：付费路径、商品结构、触发场景、漏斗转化是否变化。把指标波动和具体行为变化对应起来。

5外部因素排除

排除版本发布、活动、价格调整、买量结构、市场大盘、数据口径变更/埋点改动等外部干扰，确认根因。

6结论输出

答案优先：一句话说清主因 + 证据 + 业务影响 + 建议动作 + 信心程度。

贡献度公式

当 收入 = A × B 时，A、B 各自对收入变化的贡献可拆解为：
ΔRevenue = (A₁ - A₀) × B₀ ← A 的贡献（量的变化） + A₀ × (B₁ - B₀) ← B 的贡献（率的变化） + (A₁ - A₀) × (B₁ - B₀) ← 交互项 贡献度% = 单因子贡献 / ΔRevenue 总额

三因子（DAU × 付费率 × ARPPU）可逐层套用：先拆 DAU vs (付费率×ARPPU)，再把后者拆成付费率 vs ARPPU。

12 个常用拆解维度

维度类别具体维度
用户属性新老用户、注册天数/生命周期阶段、付费历史档位
获取来源国家/地区、买量渠道、媒体来源、自然量 vs 买量
产品环境App 版本、平台（iOS/Android）、机型/系统

数据需求清单（拆解前先备齐）

  • 收入大盘日粒度时间序列（含 DAU、付费用户数、付费金额）
  • 各拆解维度的明细（国家 / 渠道 / 版本 / 新老 / 商品）
  • 版本发布、活动、价格调整的时间节点表（用于第 5 步排除）
  • 埋点/口径变更记录（避免把数据问题当业务问题）
广告变现（IAA）全链路漏斗分析方法。核心：用 adsessionid 串联一次广告会话的完整链路，逐环节定位加载、播放、转化的损耗点。

广告全链路漏斗

1广告请求

触发广告位、向 mediation 发起请求。

2加载成功

广告素材成功返回并加载。加载成功率 = 加载成功 / 广告请求。

3展示/播放

广告成功展示给用户。播放成功率 = 播放成功 / 加载成功。

4完整观看 / 转化

用户看完或产生奖励发放。转化率 = 完成 / 播放成功。

5收益

产生 eCPM 收益。eCPM = 收益 / 展示次数 × 1000。

关键指标定义

指标定义
加载成功率加载成功次数 / 广告请求次数
播放成功率播放成功次数 / 加载成功次数
转化率完成观看次数 / 播放成功次数
eCPM广告收益 / 展示次数 × 1000，衡量单位展示价值
人均广告次数展示次数 / 活跃用户数，衡量广告频次
广告 LTV用户生命周期内累计广告收益

核心分析维度

维度说明
mediation聚合平台（如 MAX、AdMob），对比不同聚合层表现
adchannel广告渠道/网络（Meta、Google、Unity…）
arm（广告位）激励视频、插屏、Banner、开屏等广告形式
material（素材）具体广告素材，定位素材级别的转化差异
国家 / 版本 / 平台常规人群与环境维度

adsessionid：会话串联

用 adsessionid 把一次广告会话的"请求 → 加载 → 播放 → 完成 → 收益"所有事件串联起来，才能算准每一环的真实转化率，定位损耗发生在哪一步。这是广告漏斗分析的核心技术前提。

6 个常用分析视图

  • 大盘趋势：广告收益、eCPM、人均次数、填充率的时间趋势
  • 漏斗损耗：请求→加载→播放→完成各环节转化率，定位瓶颈
  • 渠道对比：不同 adchannel / mediation 的 eCPM 与填充率
  • 广告位分析：各 arm 形式的收益贡献与用户体验平衡
  • 素材分析：material 级别的转化与收益，指导素材优化
  • 人群分析：国家/版本/新老用户的广告价值差异

把分析需求拆成标准化流程。区分"活动类"与"复杂探索类"两种工作流，配合埋点准则，保证分析可复用、可追溯。

埋点准则

  • 先确认埋点再写 SQL：分析前必须确认相关行为是否有埋点、字段含义、上报时机
  • 口径以文档为准：指标定义、过滤条件参考 Confluence 真实文档，不凭经验猜
  • 关注埋点变更：埋点改动/删除会造成数据断层，分析前查变更记录
  • 明确分析单元：事件级 / 用户级 / 会话级，避免粒度混用导致重复计算

活动类分析工作流

针对运营活动、版本功能上线等有明确时间节点的分析。

1明确活动信息

活动时间、参与门槛、目标人群、预期目标指标。

2选择对比基准

活动前 vs 活动中 vs 活动后；或参与组 vs 未参与组。注意完整周期对比，避免部分期间比较。

3核心指标评估

参与率、目标指标提升、收入/留存影响，量化活动效果。

4排除干扰

剔除自然增长、其他并行活动、买量波动等因素。

5结论与建议

活动是否达标、ROI、是否值得复用/放量。

复杂探索类分析工作流

针对开放式问题（"为什么 X 下降了？""哪类用户更值钱？"），没有现成答案需要探索。

1拆解问题

把模糊问题拆成可量化的子问题，明确要回答什么、用什么证据证明。

2建立假设

基于业务理解列出可能原因（假设），按可能性和可验证性排序。

3逐一验证

用数据验证/推翻每个假设，从最可能、最易验证的开始。

4交叉下钻

对成立的假设继续多维拆解，定位到具体人群/场景/行为。

5综合结论

整合多条证据形成完整因果链，输出结论与行动建议。

6 步对话工作流

1需求澄清

用提问把模糊诉求变清晰：想看什么指标、什么时间范围、哪个产品、对比什么。给选项让用户勾选，不问开放题。

2方案确认

把理解到的分析方案复述给用户确认：分析目标、口径、维度、产出形式。确认后再往下。

3确认字段

明确用到的表、字段、埋点。引用数据口径字典，确保字段真实存在、口径正确。

4SQL 生成

基于确认的字段和口径生成 SQL，遵守各产品表名/主键差异。

5SQL 自评

生成后自查：粒度对不对、join 会不会爆炸、过滤条件全不全、口径有没有踩坑。

6解读指引

告诉用户跑出来的数据怎么看、关注哪些异常、下一步可以怎么深挖。

5 条数据管理规范

规范说明
先写后查先把 SQL 写进文件存档，再执行查询，保证可追溯、可复用
SQL 变更策略SQL 有改动时保留版本，记录改了什么、为什么改
CSV 命名规范结果 CSV 用统一命名（主题_产品_日期），方便归档检索
SQL 与 CSV 关联每份 CSV 对应记录其来源 SQL，数据与查询一一对应
UTF-8 BOM 头导出 CSV 带 UTF-8 BOM 头，避免 Excel 打开中文乱码

对话式引导原则

  • 给选项，不问开放题：用编号 + 字母（如 1A / 2B）让用户快速回复，降低输入成本
  • 一次只问一件事：分步引导，不要一次抛一堆问题
  • 能勾选不填写：尽量用选择代替手动输入
  • 语言精炼直白：10 岁小孩也能看懂，不用术语黑话
  • 每步可确认可回退：复述理解让用户确认，错了能改

SQL 高手指南（自评要点）

  • 粒度优先：写之前先想清楚结果是一行代表什么（用户/天/事件）
  • 防 join 爆炸：多对多 join 会静默倍增行数，join 后核对行数
  • 过滤完整：日期、产品、平台、测试用户排除等过滤条件不能漏
  • 口径对齐：指标定义、分子分母、去重逻辑严格对齐文档
  • 大表先限流：探索阶段先加日期/LIMIT 限制扫描量，避免全表扫描
评估数据集是否可信以用于分析/建模/仪表盘；并验证已完成的分析在分享前是否可靠。

Part A：数据质量评估

工作流

1明确质量问题和操作上下文

建立数据集代表什么、预期分析单元、下游用途。确定主键、日期列、时区假设、领域规则、业务阈值。

2选择可检查的分析路径

默认使用配套 notebook 或脚本，以便检查发现背后的确切代码。先确认 schema、粒度、样本行。

3构建紧凑的数据画像

行数、列数、类型、候选键重复率、空值率、时间戳范围。先确认粒度——许多表面问题实际是混合粒度/部分回填/重复 join。

4运行核心质量检查

完整性、唯一性、有效性、一致性、引用完整性、时效性、容量与形态。比较比率而非仅计数，按维度切分。

5形态特定检查 + 时间分布检查

事件/维度/事实/特征/实验数据各有专属检查。有历史数据时关注变化点（发布、迁移、事故）。

6调查风险 + 推荐修复

将每个问题与下游风险联系，推荐最小一组修复/监控/自动化测试。

核心检查类别

类别检查内容
完整性按列/分区/分段统计空值率；意外空字符串或哨兵值；必需列填充率
唯一性完全重复行、重复主键、重复复合键
有效性类型符合性；ID/邮箱/URL/枚举/时间戳格式；度量和日期范围检查
一致性跨字段规则、单位/货币一致性、状态-时间戳对齐
引用完整性父子键覆盖、孤儿记录、意外多对多 join、断裂 SCD join
时效性新鲜度延迟、缺失最近分区、无法解释的历史重写/回填
容量和形态行数漂移、不同值计数漂移、分布漂移、新增/消失类别

严重程度分类

级别定义
Critical中断可信分析、核心 join、生产仪表盘或关键决策
High实质性偏向下游决策（大范围空值激增、类别漂移、泄漏）
Medium局部或可解释问题，需文档记录或监控
Low表面不一致、预期稀疏性或已知边缘情况

Part B：分析验证（QA）

常见陷阱

  • Join 爆炸：多对多 join 静默地倍增行
  • 幸存者偏差：仅包括今天存在的实体
  • 不完整期间比较：部分期间 vs 完整期间
  • 分母偏移：比较组之间合格人群变化
  • 平均值的平均值：未加权平均预计算的均值
  • 时区不匹配：不同时间戳约定或截止点
  • 辛普森悖论：聚合和分段级别趋势冲突

置信度评级

评级含义
Ready to share方法论健全，关键计算已验证，注意事项清晰
Share with caveats方向性可用，但必须传达特定限制
Needs revision实质性错误、不受支持的声明或方法论问题
诊断指标为何变更或与预期不同，并生成面向领导层的 KPI 更新。

Part A：指标诊断

1定义诊断问题

框定什么变了、什么比较能证明它。明确时间窗口、人群、粒度、诊断类型（变动/集中度/对账）。

2验证指标定义和来源

确认定义、粒度、聚合逻辑、过滤、join、排除、新鲜度、血缘。至少检查一个业务面 + 一个低层来源。

3建立指标模式

量化相关期间和范围内的指标，重现比较。模式的大小、时间、范围验证前不要找原因。

4选择诊断计划

从指标操作逻辑、业务上下文、源形态中选驱动因素维度。优先业务可监控/可行动的因素。

5分解并验证驱动因素

用最强证据衡量每个驱动因素。比率指标看分子/分母/两者；区分混合效应 vs 段内绩效效应。

6陈述含义和后续行动

以诊断问题的答案开头，陈述最强驱动因素 + 证据 + 业务影响 + 信心程度 + 下一步。

解释模式

场景方法
指标变更焦点窗口 vs 基线，对分段贡献排名，检查混合偏移 vs 段内变动
尖峰/回归/事故确定开始/峰值/恢复、分布形态、受影响切片、广泛 vs 局部退化
最大贡献者定义"最大"，对实体排名，比较总量份额和变化，找主要变动者/进入者/退出者
对账对齐定义/过滤/粒度/分子/分母，量化解释差距的组成部分

Part B：KPI 报告

报告结构

**Status**: [On track / At risk / Ahead of plan] **Current**: [Value] vs. Target: [Value] ([+/-X%, pacing]) **Driver**: [What's driving performance] **Implication**: [What the team should do]

呈现标准

  • 为略读的高管写作：以答案开头，然后是证据
  • 使用业务可读的数字：123k (+8% w/w, +19% m/m)
  • 用指标证据替代通用形容词（"强劲"、"健康"）
  • 将注意事项放在它们影响的声明附近
  • 仅在有助于优先行动时使用交通信号灯状态

Part A：KPI 框架设计

1明确决策和操作上下文

理解指标支持的决策、审查上下文、谁对结果采取行动。

2收集证据再推荐指标

了解目标、当前状态、受众、约束、风险、现有定义、基线/目标上下文。

3生成更广泛的候选集

创建候选结果、驱动因素、护栏指标，每个有明确定义和与决策的合理联系。

4比较和选择指标

按 6 标准筛选，推荐 1-3 个主要 KPI + 每个 1-2 个驱动因素 + 1-2 个护栏。

5设定目标

自上而下（基准/历史/竞品）vs 自下而上（团队能做什么）。比较雄心与现实影响力。

整体-部分stacked bar分母明确；少数切片才用 pie
分布/扩散histogram揭示形态的数值 bin；比较组用 boxPlot
变量间关系scatter数值 x/y；目标 12-20 个有意义的点
密集 2D 模式heatmap矩阵形状或强度；点级变化切 scatter
加性桥接waterfall驱动因素干净加到最终值；否则排序 bar
有序阶段funnel仅用于有序单系列阶段

视觉设计标准

排版和颜色

  • 跨图表单一字体系列；白色/近白背景、安静灰网格线、深炭文本
  • 不仅依赖颜色——用色调、填充、标记、线条样式、直接标签、排序
  • 调色板策略：单根色优先 / 硬性双根色上限 / 宽松多类别（≤5）

选择规则

  • 从分析问题开始，不是最喜欢的图表类型
  • 图表用于形态和比较；表格用于精确查找
  • 不交付动力不足的趋势图（<8点）或散点图（<12点）
  • pie / Pareto / waterfall / Likert 保留为变体而非默认

质量门槛

  • 图表选择遵循分析问题，而非视觉偏好
  • 形式匹配比较；比例诚实一致
  • 每个图表有可见标题 + 带必要上下文的副标题
  • 正/负状态用色调 + 符号标签，不仅是颜色
构建精美的分析报告、仪表盘和产品/业务分析。

Part A：报告构建

1定义报告任务

陈述问题、决策、受众、范围、时间框架、比较基线。选一种受众：产品利益相关者 / 技术。

2收集并界定证据

清点源数据、指标定义、分母、假设、注意事项、notebook/SQL。草拟声明前解决歧义。

3提炼报告主干

答案优先：问题 → 答案 → 指标口径 → 发现+证据 → 验证 → 注意事项 → 下一步。

4规划读者面向的结构

有序主段落 + 可见标题。定量发现默认可视化，表格用于精确查找。

5设计可视化和表格

每个视觉支持特定声明，规划相邻解释段落。

报告标准

  • 利益相关者报告以执行摘要开启（2-4 个带粗体主题句的要点）
  • 以结论开启主段落，而非设置性散文
  • 每个重要数字/可视化与通俗英语解释配对
  • 将原始数字放入上下文：基准百分比、基线比较、历史范围
  • 在叙述中用速记大数字（899k, 10.9k, 1.2M）

指标模型分类

类型测量对象
Reach 触达谁/什么在使用、渗透率、激活、覆盖

布局原则

  • 从摘要到详情：关键状态/KPI → 随时间变动 → 分解 → 详情表格
  • 默认视图在交互前回答主要受众问题
  • 过滤器少、有意义且正确工作
  • 卡片、图表、表对账一致
